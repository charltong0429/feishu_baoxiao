**日期：** 2026-06-30

## 一、项目目标

员工通过飞书小程序，用**自然语言描述**或**拍照上传票据**两种方式提交报销信息。系统结合公司报销规定自动校验，通过后在飞书中直接发起已有的审批流程，减少手动填表负担。

## 三、核心数据流

### 3.1 自然语言输入流程

```
员工输入: "昨天打车去客户那里，花了68块，滴滴出行"
    ↓
POST /api/extract（文字模式）
    ↓ 调用 OpenRouter（用用户自己的 Key）
    ↓ LLM 返回结构化 JSON：
      {
        "date": "2026-06-29",
        "amount": 68,
        "type": "交通费",
        "vendor": "滴滴出行",
        "reason": "拜访客户"
      }
    ↓
POST /api/validate 对比公司规定
    → 软提示: "交通费单次上限 100 元，本次符合规定 ✓"
    ↓
员工在预览页确认 → POST /api/submit → 飞书审批
```

### 3.2 OCR 票据识别流程

```
员工拍照/上传发票图片
    ↓
POST /api/extract（图片模式，multipart/form-data）
    ↓ 图片 + Prompt 发给 OpenRouter 多模态模型
    ↓ LLM 识别返回同样的结构化 JSON（与 3.1 格式一致）
    ↓
（与 3.1 合并，走同样的 validate → submit 流程）
```

> **设计说明：** 自然语言和 OCR 共用同一个 `/api/extract` 接口，通过请求体中的 `content_type` 字段区分文字/图片，减少重复代码。

### 3.3 API Key 首次设置流程

```
首次打开小程序 → GET /api/key 检测无 Key → 跳转设置页
    ↓
员工填入 OpenRouter API Key → POST /api/key
    ↓
后端：
  1. 通过飞书 OAuth code 换取 open_id
  2. 生成随机 IV
  3. AES-256-GCM 加密 Key
  4. 存入云数据库（open_id, encrypted_key, iv）
    ↓
后续请求：后端自动解密取用，前端不感知
```

## 四、技术栈

### 前端（飞书小程序）

| 组件 | 技术 | 说明 |
|------|------|------|
| 小程序框架 | 原生飞书小程序（TTSS） | 官方支持，审批 API 对接最顺畅 |
| 图片上传 | `tt.chooseImage` | 飞书内置相机/相册 API |
| 状态管理 | 原生 Page data | 首版不引入额外框架 |

### 后端（Node.js）

| 模块 | 技术 | 说明 |
|------|------|------|
| Web 框架 | Express.js | 轻量，适合 API Server |
| AI 调用 | `openai` SDK（base_url 指向 OpenRouter） | 兼容 OpenAI 接口格式 |
| 加密 | Node.js `crypto`（AES-256-GCM） | 加密存储用户 API Key |
| 数据库 ORM | Prisma | 类型安全，迁移管理方便 |
| 数据库 | 云数据库 PostgreSQL | 托管服务，无需自运维 |
| 身份验证 | 飞书 OAuth2（`code` 换 `open_id`） | 每次请求验证用户身份 |
| 规定解析 | `pdf-parse` + `mammoth`（Word） | 管理员上传后解析为结构化规则 |

### `users` 表

| 字段 | 类型 | 说明 |
|------|------|------|
| `open_id` | VARCHAR (PK) | 飞书用户唯一标识 |

| 字段 | 类型 | 说明 |

| `id` | SERIAL (PK) | 规则 ID |

| `description` | TEXT | 规则说明文字 |
| `updated_at` | TIMESTAMP | 最近更新时间 |

## 六、安全设计

1. **API Key 加密存储：** AES-256-GCM，每个用户独立 IV，后端负责加解密，前端永远不接触原始 Key。
2. **身份验证：** 每个后端请求必须携带飞书 OAuth `code`，后端验证后才处理，防止越权访问他人 Key。
3. **图片不持久化：** OCR 识别完成后图片不写入磁盘，仅在内存中处理后传给 LLM。
4. **报销规定只读：** 普通员工接口只能读取规则，修改规则需管理员身份（通过飞书角色判断）。

## 七、小程序页面结构

```
pages/
  index/          ← 首页（选择：自然语言 或 OCR）
  input/          ← 自然语言输入页
  ocr/            ← 拍照/上传页
  preview/        ← 预览确认页（含软提示）
  success/        ← 提交成功页（含飞书审批跳转）
  settings/       ← API Key 设置页
```

## 八、后端目录结构

```
src/
  routes/
    extract.js    ← /api/extract（自然语言 + OCR 共用）
    validate.js   ← /api/validate
    submit.js     ← /api/submit
    key.js        ← /api/key
  services/
    llm.js        ← OpenRouter 调用封装
    feishu.js     ← 飞书 OAuth + 审批 API 封装
    crypto.js     ← AES-256-GCM 加解密
    rules.js      ← 报销规定比对逻辑
  prisma/
    schema.prisma ← 数据库 Schema
```
